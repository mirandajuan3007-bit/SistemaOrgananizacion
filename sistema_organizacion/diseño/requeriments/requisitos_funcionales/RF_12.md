# RF_12 — Consulta, búsqueda y dashboard operativo

## por qué existe este requisito en este sistema

Otra necesidad central del cliente fue saber qué proyectos tienen, en qué estado están y qué está pasando con cada expediente sin tener que revisar carpetas, correos y hojas de cálculo. Esa necesidad se divide en tres capacidades funcionales concretas: listar, buscar y visualizar el estado general.

Si el sistema no permite encontrar rápidamente un proyecto, un participante o un contrato, la información seguirá siendo técnicamente almacenada pero operativamente inútil. Si no existe una vista de control general, la dirección seguirá sin visibilidad real de la operación.

---

## descripción del requisito

El sistema debe permitir consultar información operativa mediante listados, filtros, búsquedas y dashboards resumidos.

El sistema debe permitir:

- listar proyectos, participantes, contratos, documentos y pagos
- buscar por nombre, folio, proyecto, estado u otros criterios relevantes
- filtrar por unidad, responsable, rango de fechas o estatus
- visualizar dashboards de control con indicadores clave
- acceder al detalle desde cualquier resultado de búsqueda o tablero

---

## actores involucrados

- secretaría
- director o responsable académico
- administrador del sistema

---

## precondiciones

- deben existir registros operativos cargados en el sistema
- el usuario debe tener permisos para consultar la información solicitada

---

## flujo principal

1. El usuario abre un listado o el dashboard principal.
2. El sistema carga indicadores generales y resultados iniciales.
3. El usuario aplica filtros o escribe un criterio de búsqueda.
4. El sistema actualiza resultados.
5. El usuario selecciona un registro y navega a su detalle.

---

## flujos alternos y excepciones

### búsqueda sin resultados

1. El usuario ingresa un criterio no encontrado.
2. El sistema informa que no hay coincidencias y sugiere limpiar filtros o refinar búsqueda.

### acceso restringido a ciertos datos

1. El usuario visualiza un dashboard o listado.
2. El sistema solo muestra la información permitida por su rol.

### dashboard con datos históricos

1. El usuario selecciona un periodo específico.
2. El sistema recalcula indicadores para ese rango.

---

## reglas de negocio asociadas

- los resultados de búsqueda deben respetar unidad y permisos del usuario
- los indicadores del dashboard deben calcularse a partir de datos reales del sistema y no capturarse manualmente
- todo resultado de búsqueda debe permitir navegación al detalle cuando el usuario tenga permiso

---

## cómo se mide este requisito en el sistema

El requisito se considera cumplido si:

- el usuario puede encontrar rápidamente proyectos, contratos o participantes específicos
- el dashboard muestra información útil para decidir y priorizar trabajo
- desde los resultados puede saltarse al expediente o contrato correcto
- la operación ya no depende de revisar Excel o OneDrive para saber el estado general

---

## cómo se traduce en el sistema

### en el frontend

- existen listados con paginación y filtros
- existe una barra de búsqueda o buscador contextual
- existe un dashboard inicial con tarjetas, gráficas o tablas resumidas

### en el backend

- existen endpoints de consulta con filtros y paginación
- existen servicios para calcular métricas e indicadores agregados
- la búsqueda respeta permisos y contexto de unidad

### en la base de datos

- las tablas principales permiten consultas filtradas y agregaciones
- existen índices en campos de búsqueda frecuente como folio, nombre, estado o fechas

---

## cómo se evidencia que se cumple

- el director puede abrir el dashboard y ver cuántos contratos están observados, cuántos expedientes están incompletos y cuántos pagos siguen pendientes
- la secretaría puede ubicar rápidamente a un participante por nombre o folio
- desde cualquier listado se puede entrar al detalle correcto sin navegación compleja

---

## cómo se representa en diseño

### en casos de uso

Debe existir el caso de uso "buscar expediente o contrato" y "consultar dashboard operativo".

### en wireframes

Debe existir diseño del dashboard principal y de los listados filtrables.

### en el modelo analítico

Deben definirse los indicadores clave que alimentan el dashboard: pendientes, observados, listos para contrato, en pago y cerrados.
