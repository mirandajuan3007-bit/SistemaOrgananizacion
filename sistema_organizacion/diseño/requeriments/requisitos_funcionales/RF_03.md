# RF_03 — Registro y asignación de participantes al proyecto

## por qué existe este requisito en este sistema

Después de que el proyecto es aceptado, la operación real del módulo comienza cuando se identifica qué personas van a participar y bajo qué esquema. En el caso actual, el módulo principal es honorarios, por lo que hay personas que deben entregar documentación para contrato y posteriormente para pago.

Si el sistema no permite registrar formalmente a los participantes y relacionarlos con un proyecto, no puede existir expediente digital, ni control de documentos, ni seguimiento de contrato, ni trazabilidad de pagos. El participante es la unidad humana sobre la que gira el flujo documental y administrativo.

---

## descripción del requisito

El sistema debe permitir registrar participantes dentro de un proyecto y asociarlos con su tipo de participación, sus datos personales y su estado administrativo.

El sistema debe permitir:

- dar de alta un participante en un proyecto
- capturar sus datos personales y fiscales necesarios para la operación
- definir su tipo de participación
- consultar todos los participantes de un proyecto
- editar su información cuando haya correcciones válidas
- marcarlo como activo, observado, incompleto, listo para contrato, en pago o cerrado

---

## actores involucrados

- secretaría
- director o responsable académico
- investigador o participante, de forma indirecta cuando consulta su expediente

---

## precondiciones

- debe existir un proyecto activo
- el usuario debe tener permisos para operar participantes

---

## flujo principal

1. La secretaría abre el detalle de un proyecto.
2. Ingresa a la sección de participantes.
3. Selecciona agregar participante.
4. Captura datos personales, fiscales y de contacto.
5. Indica el tipo de participación y su contexto operativo.
6. El sistema valida datos obligatorios y consistencia básica.
7. El sistema crea el registro del participante dentro del proyecto.
8. El sistema crea automáticamente su expediente inicial.

---

## flujos alternos y excepciones

### participante ya existente en el sistema
 
1. El usuario captura RFC o correo de una persona ya registrada.
2. El sistema detecta coincidencia.
3. En lugar de duplicar, ofrece reutilizar los datos base y asociar al participante al nuevo proyecto según las reglas definidas.

### corrección de datos

1. La secretaría detecta un dato incorrecto.
2. Edita el participante.
3. El sistema conserva registro de qué dato cambió y cuándo.

### baja lógica del participante

1. El usuario necesita retirar al participante del flujo.
2. El sistema no lo elimina físicamente.
3. Lo marca con estado inactivo o cancelado preservando su historial.

---

## reglas de negocio asociadas

- un participante debe estar asociado a al menos un proyecto
- un participante no debe duplicarse innecesariamente si ya existe con el mismo identificador fiscal o institucional
- la baja del participante no debe borrar contratos, documentos o pagos históricos
- el estado del participante debe reflejar su avance dentro del flujo administrativo

---

## cómo se mide este requisito en el sistema

El requisito se considera cumplido si:

- el alta de participantes puede realizarse desde el detalle del proyecto
- cada participante tiene expediente y estado operativo visibles
- el sistema reduce duplicados mediante validaciones o coincidencias previas
- los cambios a los datos del participante quedan registrados

---

## cómo se traduce en el sistema

### en el frontend

- existe un formulario de alta de participante dentro del proyecto
- existe una lista filtrable de participantes por proyecto y estado
- existe una vista de detalle individual del participante

### en el backend

- existe un módulo o servicio de participantes
- el backend valida duplicados razonables y relación obligatoria con proyecto
- al crear un participante, el backend inicializa su expediente y estados base

### en la base de datos

- existe una tabla `participantes`
- existe una relación entre `participantes` y `proyectos`
- existe un campo de estado administrativo del participante

---

## cómo se evidencia que se cumple

- puede registrarse un participante y verlo inmediatamente dentro del proyecto
- al intentar registrar nuevamente a la misma persona, el sistema alerta posible duplicidad
- el participante conserva historial aunque deje de estar activo
- desde el proyecto puede verse quiénes participan y en qué estado están

---

## cómo se representa en diseño

### en casos de uso

Debe existir el caso de uso "registrar participante en proyecto".

### en el modelo de datos

La entidad `Participante` debe relacionarse con `Proyecto`, `Expediente`, `Contrato`, `Documento` y `Pago`.

### en el flujo operativo

El participante debe aparecer como paso inmediatamente posterior al proyecto aceptado.
