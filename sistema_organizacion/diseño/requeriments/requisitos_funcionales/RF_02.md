# RF_02 — Registro y gestión de proyectos

## por qué existe este requisito en este sistema

El proceso operativo descrito por el cliente inicia cuando llega una solicitud o un trabajo para la universidad y se construye una estimación, un presupuesto o un acuerdo que finalmente da origen a un proyecto aceptado. A partir de ese momento, todo lo demás depende de ese proyecto: personas, contratos, documentos, pagos, observaciones y cierre.

Si el proyecto no existe formalmente como entidad dentro del sistema, no hay forma de organizar de manera consistente lo que sucede después. El sistema volvería a caer en carpetas sueltas, nombres inconsistentes y seguimientos hechos por memoria o Excel.

Este requisito es funcional porque el personal administrativo debe poder dar de alta un proyecto, consultarlo, modificar sus datos y usarlo como contenedor operativo del resto del flujo.

hay que hacer esta parte como un modulo ya que solo es una parte del sistema es el registro y la gestion de los proyectos.
---

## descripción del requisito

El sistema debe permitir registrar proyectos y administrarlos como unidad operativa principal del módulo.

Cada proyecto debe concentrar al menos:

- nombre o título del proyecto
- cliente o entidad solicitante
- descripción resumida del trabajo
- fechas relevantes
- estado general del proyecto
- responsable académico o administrativo
- monto aprobado o referencia presupuestal
- unidad académica a la que pertenece

estos datos son los minimos indispensables.

El sistema debe permitir:

- crear un proyecto nuevo
- editar sus datos base
- consultar su detalle completo
- cambiar su estado general
- listar proyectos por estado, fecha, responsable o cliente
- cerrar el proyecto cuando su operación haya concluido

---

## actores involucrados

- secretaría
- director o responsable académico
- administrador del sistema

---

## precondiciones

- el usuario debe haber iniciado sesión y tener permisos operativos
- la unidad académica debe existir y estar activa
- el proyecto debe pertenecer a una unidad específica desde su creación

---

## flujo principal

1. La secretaría entra al módulo de proyectos.
2. Selecciona la opción de crear proyecto.
3. Captura los datos básicos del proyecto.
4. El sistema valida que los campos obligatorios estén completos.
5. El sistema genera el registro del proyecto con estado inicial.
6. El sistema muestra la pantalla de detalle del proyecto recién creado.
7. Desde esa pantalla, el usuario puede comenzar a registrar participantes y documentos asociados.

---

## flujos alternos y excepciones

### proyecto duplicado

1. El usuario intenta registrar un proyecto con el mismo nombre, cliente y periodo que uno existente.
2. El sistema detecta coincidencia alta y advierte posible duplicado.
3. El usuario decide cancelar o continuar con justificación.

### modificación de datos del proyecto

1. El usuario abre el proyecto.
2. Selecciona editar.
3. Cambia campos permitidos. (con esto quiero decir que no puedes cambiar cualquier campo de la informacion del proyecto)
4. El sistema registra la modificación y conserva trazabilidad. (con conservar la trazabilidad me refiero a que hya un registro de los cambios que se hicieron)

### cierre del proyecto

1. El usuario solicita cerrar el proyecto.
2. El sistema valida que no existan contratos abiertos o pagos pendientes críticos.
3. Si la validación es correcta, cambia el proyecto a cerrado.
4. Si no lo es, informa qué pendientes impiden el cierre.

---

## reglas de negocio asociadas

- todo contrato, participante, documento y pago debe pertenecer a un proyecto
- un proyecto no puede cerrarse si tiene elementos críticos pendientes definidos por la operación
- el proyecto debe tener un estado general visible para seguimiento institucional
- el identificador interno del proyecto debe ser único dentro de la unidad académica

---

## cómo se mide este requisito en el sistema

El requisito se considera cumplido si:

- el personal puede crear un proyecto sin depender de soporte técnico
- todos los registros operativos del módulo quedan asociados a un proyecto válido
- es posible consultar la lista de proyectos con filtros operativos útiles
- el sistema impide cerrar proyectos con pendientes incompatibles con el cierre

---

## cómo se traduce en el sistema

### en el frontend

- existe una vista de listado de proyectos
- existe un formulario de alta y edición
- existe una vista de detalle con pestañas o secciones para participantes, contratos, documentos y pagos

### en el backend

- existe un módulo de proyectos con endpoints para crear, consultar, actualizar y cerrar proyectos
- existe validación de reglas de cierre antes de cambiar el estado

### en la base de datos

- existe una tabla `proyectos`
- las tablas del módulo referencian a `proyectos` mediante clave foránea
- existe un campo de estado general y un identificador único por unidad

---

## cómo se evidencia que se cumple

- puede darse de alta un proyecto y luego usarlo como origen de contratos y expedientes
- el sistema muestra todos los proyectos activos y cerrados con filtros operativos
- al intentar cerrar un proyecto con pendientes, el sistema lo bloquea y explica por qué
- el detalle del proyecto concentra toda la operación vinculada a él

---

## cómo se representa en diseño

### en casos de uso

Debe existir el caso de uso "registrar proyecto" y "consultar estado de proyecto".

### en el modelo de datos

La entidad `Proyecto` debe ser agregadora de `Participante`, `Contrato`, `Documento` y `Pago`.

### en la navegación

Debe existir una entrada principal al módulo de proyectos y una vista de detalle como punto de entrada al resto del flujo.

### en diagramas de flujo

El proyecto debe aparecer como el punto de inicio de la operación administrativa posterior a la aceptación del trabajo.
