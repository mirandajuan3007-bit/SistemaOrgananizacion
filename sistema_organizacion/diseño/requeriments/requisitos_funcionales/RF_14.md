# RF_14 — Consulta del historial de cambios de entidades operativas

## por qué existe este requisito en este sistema

Una parte del problema del cliente no es solo que el sistema guarde evidencia técnica de los cambios, sino que la operación pueda verla sin reconstruirla manualmente desde correos, documentos y memoria personal. La secretaría necesita saber qué pasó con un contrato, quién hizo una observación, cuándo se corrigió un documento y cuál fue el último cambio importante en el expediente.

Eso ya no es una propiedad transversal del sistema, sino una función visible para el usuario. Por eso conviene separarlo del requisito no funcional de auditabilidad.

---

## descripción del requisito

El sistema debe permitir que los usuarios autorizados consulten el historial de cambios relevantes de contratos, documentos, participantes, pagos y expedientes desde sus pantallas de detalle.

La consulta del historial debe mostrar, al menos:

- fecha y hora del evento
- usuario u origen que realizó la acción
- tipo de cambio realizado
- estado anterior y estado nuevo cuando aplique
- observación asociada o contexto breve del cambio cuando exista

---

## actores involucrados

- secretaría
- director, con permisos de consulta según alcance definido
- administrador del sistema

---

## precondiciones

- debe existir la entidad sobre la cual se quiere consultar el historial
- el usuario debe tener permisos para consultar esa entidad y su historial
- el sistema debe contar con registros de auditoría generados por la operación previa

---

## flujo principal

1. El usuario abre el detalle de una entidad, por ejemplo un contrato o expediente.
2. El usuario ingresa a la sección de historial.
3. El sistema consulta los eventos asociados a esa entidad.
4. El sistema muestra la línea de tiempo ordenada cronológicamente.
5. El usuario revisa qué cambió, cuándo ocurrió y quién realizó cada acción.

---

## flujos alternos y excepciones

### historial sin eventos visibles

1. El usuario abre la sección de historial.
2. La entidad aún no tiene cambios registrados más allá de su creación, o no existen eventos consultables.
3. El sistema muestra el mensaje correspondiente sin generar error.

### acceso sin permisos suficientes

1. Un usuario intenta abrir el historial de una entidad a la que no debería acceder.
2. El sistema bloquea la consulta o restringe el detalle mostrado según la política de acceso.

### historial extenso

1. Una entidad tiene muchos eventos acumulados.
2. El sistema pagina o agrupa el historial para mantenerlo legible.

---

## reglas de negocio asociadas

- solo usuarios autorizados pueden consultar historial
- la consulta del historial no debe permitir modificar los registros mostrados
- el historial debe presentarse en un orden temporal claro
- la información mostrada debe ser suficiente para entender el cambio sin exponer datos sensibles innecesarios

---

## cómo se mide este requisito en el sistema

El requisito se considera cumplido si:

- un usuario autorizado puede consultar el historial de una entidad desde su pantalla de detalle
- la secuencia mostrada permite entender los cambios relevantes del caso
- el sistema distingue claramente entre el estado anterior y el estado nuevo cuando hubo transición
- la consulta del historial respeta permisos y alcance de acceso

---

## cómo se traduce en el sistema

### en el frontend

- existe una sección o pestaña de historial en las pantallas de detalle relevantes
- los eventos se muestran como línea de tiempo o lista cronológica
- la interfaz permite leer quién hizo el cambio, cuándo y qué cambió

### en el backend

- existe un endpoint de lectura de historial por entidad
- el backend filtra los eventos por entidad e identidad del usuario autorizado
- el backend transforma los registros de auditoría en una respuesta legible para la interfaz

### en la base de datos

- la información consultada proviene de la tabla `audit_log` o equivalente
- los registros se relacionan con la entidad mediante `entity_type` y `entity_id`

---

## cómo se evidencia que se cumple

- desde un contrato puede consultarse la secuencia de cambios desde su creación hasta su estado actual
- desde un documento puede verse cuándo fue cargado, observado, corregido o aprobado
- un usuario sin permiso no puede consultar el historial de entidades fuera de su alcance
- la vista de historial permite entender el caso sin revisar correos o archivos externos para reconstruirlo

---

## cómo se representa en diseño

### en casos de uso

Debe existir el caso de uso "consultar historial de cambios".

### en la interfaz

Debe existir una sección de historial en el detalle de contrato, documento, participante, pago o expediente cuando aplique.

### en la arquitectura

La consulta del historial debe apoyarse en el módulo de auditoría, pero representarse como una función de lectura para usuarios autorizados.
