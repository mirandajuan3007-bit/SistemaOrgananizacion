# RF_09 — Registro y control del pago de honorarios

## por qué existe este requisito en este sistema

El flujo del módulo no termina cuando el contrato está resuelto. El cliente explicó que todavía falta la parte económica: factura, evidencia, cheque, complemento de pago y control de si una persona cobró o no cobró a tiempo. Esa parte también hoy se dispersa y genera problemas posteriores, especialmente cuando la persona quiere volver a ser contratada.

Si el sistema no incorpora el pago como parte formal del expediente, se corta el flujo justo antes del cierre real del proceso.

---

## descripción del requisito

El sistema debe permitir registrar y consultar el avance del pago de honorarios asociado a un participante y a su contrato.

El sistema debe permitir:

- registrar factura o referencia de cobro
- registrar evidencias necesarias para pago
- registrar emisión o entrega del cheque o medio equivalente
- registrar complemento de pago cuando aplique
- distinguir pagos pendientes, en proceso, entregados, cobrados o incompletos
- mostrar si existen pendientes que impidan futuras contrataciones

---

## actores involucrados

- secretaría
- personal administrativo de apoyo contable
- director, en consulta de seguimiento

---

## precondiciones

- debe existir un contrato en estado compatible con pago
- deben existir las evidencias mínimas requeridas según las reglas de la operación

---

## flujo principal

1. La secretaría abre el expediente del participante.
2. El sistema indica que el contrato ya puede pasar a fase de pago.
3. El usuario registra factura, evidencia y datos del proceso de pago.
4. El sistema actualiza el estado del pago.
5. Si se entrega cheque o equivalente, se registra esa entrega.
6. Si se recibe complemento de pago, se marca como completo.
7. El expediente refleja el estado final o pendiente del pago.

---

## flujos alternos y excepciones

### cheque no cobrado

1. Se registra emisión o entrega del cheque.
2. El sistema conserva el estado como entregado o pendiente de cobro según la regla elegida.
3. La operación puede seguir monitoreando ese caso.

### complemento de pago faltante

1. El pago se realizó, pero falta complemento.
2. El sistema marca el expediente como administrativamente incompleto.
3. La persona puede quedar señalada con pendiente para futuras contrataciones.

### evidencia insuficiente

1. El usuario intenta avanzar a pago.
2. El sistema detecta que faltan pruebas o evidencias requeridas.
3. Bloquea el avance y explica qué falta.

---

## reglas de negocio asociadas

- un pago debe estar asociado a contrato y participante
- el pago no debe avanzar si faltan requisitos críticos previos
- el complemento de pago puede ser obligatorio para considerar el ciclo totalmente cerrado
- los pendientes económicos deben reflejarse en el expediente y en alertas futuras

---

## cómo se mide este requisito en el sistema

El requisito se considera cumplido si:

- puede consultarse el estado de pago de cualquier participante desde su expediente
- el sistema distingue claramente entre pago realizado y pago administrativamente completo
- el sistema puede identificar personas con pendientes que impacten futuras contrataciones

---

## cómo se traduce en el sistema

### en el frontend

- existe una sección de pagos dentro del expediente
- existen formularios para registrar factura, evidencia, entrega y complemento
- existen etiquetas visibles del estado de pago

### en el backend

- existe lógica para validar si el contrato puede pasar a pago
- existe servicio de pagos con cambios de estado y reglas documentales asociadas

### en la base de datos

- existe una tabla `pagos`
- puede existir tabla o campos para `facturas`, `complementos_pago` y `evidencias`
- la tabla se relaciona con `contratos` y `participantes`

---

## cómo se evidencia que se cumple

- puede saberse si un participante ya cobró, si falta complemento o si el caso sigue en proceso
- al faltar evidencia o complemento, el sistema lo refleja como pendiente visible
- el historial de pago queda integrado al expediente del participante

---

## cómo se representa en diseño

### en casos de uso

Debe existir el caso de uso "registrar pago de honorarios".

### en el modelo de datos

La entidad `Pago` debe relacionarse con `Contrato`, `Participante` y sus evidencias.

### en diagramas de flujo

Debe modelarse el paso contrato resuelto → evidencia → pago → complemento → cierre.
