# RF_04 — Definición y control de documentos requeridos

## por qué existe este requisito en este sistema

Uno de los problemas más claros que describió el cliente es que a las personas se les solicita documentación diversa y luego se pierde el control de qué ya entregaron, qué sigue faltando, qué fue observado y qué ya está correcto. Hoy esa memoria vive en correos, listas informales y revisión manual.

El sistema necesita convertir esa exigencia documental en una estructura explícita. Si no existe una definición clara de documentos requeridos por tipo de participante o tipo de trámite, el sistema no puede decir qué falta ni puede decidir si alguien está listo para contrato o pago.

---

## descripción del requisito

El sistema debe permitir definir, asociar y controlar los documentos requeridos para cada participante y para cada fase operativa.

El sistema debe permitir:

- definir catálogos de tipos de documento
- indicar cuáles documentos son obligatorios para un flujo determinado
- marcar documentos como opcionales, condicionales o requeridos
- visualizar por participante qué documentos hacen falta
- registrar el estado documental de cada requisito

---

## actores involucrados

- secretaría
- administrador funcional del sistema
- participante, cuando consulta qué le falta

---

## precondiciones

- debe existir un participante asociado a un proyecto
- deben existir tipos de documento definidos en el catálogo del sistema

---

## flujo principal

1. La secretaría o administrador define los tipos de documento requeridos para el flujo.
2. El sistema asocia esos documentos al expediente del participante según su tipo de participación.
3. El expediente muestra una lista de requisitos documentales.
4. Conforme llegan documentos, cada requisito cambia de estado.
5. El sistema recalcula automáticamente si el expediente está completo o incompleto.

---

## flujos alternos y excepciones

### documento condicional

1. El usuario configura un documento que solo aplica en ciertos casos.
2. El sistema lo solicita únicamente cuando la condición se cumple.

### cambio en política documental

1. La operación cambia qué documentos se exigen.
2. El catálogo se actualiza.
3. El sistema aplica la nueva regla a expedientes nuevos y, si se decide, a expedientes abiertos.

### exención de requisito

1. La secretaría marca un requisito como exceptuado para un caso concreto.
2. El sistema registra quién autorizó la excepción y por qué.

---

## reglas de negocio asociadas

- los documentos requeridos deben depender del tipo de participante o del flujo administrativo
- un expediente no puede marcarse completo si conserva requisitos obligatorios pendientes
- las excepciones documentales deben quedar justificadas
- los catálogos de documentos deben ser reutilizables entre expedientes similares

---

## cómo se mide este requisito en el sistema

El requisito se considera cumplido si:

- el sistema puede mostrar para cualquier participante la lista exacta de documentos faltantes
- el estado de completitud se calcula automáticamente
- es posible distinguir entre documentos obligatorios, opcionales y exceptuados
- la operación puede ajustar el catálogo documental sin rehacer expedientes manualmente uno por uno

---

## cómo se traduce en el sistema

### en el frontend

- existe una pantalla de catálogo de tipos de documento
- en el expediente se muestra un checklist documental con estados
- existe indicador visible de expediente completo o incompleto

### en el backend

- existe lógica para asignar requisitos documentales a un participante según reglas definidas
- existe recálculo del estado documental cuando cambia un requisito

### en la base de datos

- existe una tabla de `tipos_documento`
- existe una tabla de `requisitos_documentales`
- existe una tabla relacional entre expediente y requisito documental con su estado

---

## cómo se evidencia que se cumple

- al abrir un expediente, el sistema indica exactamente qué documentos faltan
- al marcar un documento como recibido o aprobado, el checklist cambia inmediatamente
- al completar todos los requeridos, el expediente cambia a estado documental completo
- las excepciones documentales quedan visibles y justificadas

---

## cómo se representa en diseño

### en casos de uso

Debe existir el caso de uso "definir documentos requeridos" y "consultar documentos faltantes".

### en el modelo de datos

Debe representarse la relación entre `TipoDocumento`, `RequisitoDocumental` y `Expediente`.

### en la interfaz

Debe existir un checklist documental como componente principal del expediente.
