# RF_05 — Expediente digital del participante

## por qué existe este requisito en este sistema

El cliente describió que actualmente la información de una persona se dispersa entre carpetas, correos, comentarios en Word, archivos sueltos y seguimiento manual. El sistema que se quiere construir busca precisamente lo contrario: un expediente digital que concentre todo lo que importa sobre cada participante.

El expediente digital es la vista operativa que le da sentido al sistema. Sin expediente, la información seguiría fragmentada y la secretaría tendría que abrir varias herramientas para saber en qué situación está una persona.

---

## descripción del requisito

El sistema debe generar y mantener un expediente digital por participante dentro de cada proyecto, donde se concentre toda su información operativa relevante.

El expediente debe reunir al menos:

- datos generales del participante
- estado documental
- historial del contrato
- observaciones activas o históricas
- archivos asociados
- estado de pago o cierre
- línea de tiempo de eventos relevantes

---

## actores involucrados

- secretaría
- director, con acceso de consulta
- participante, con acceso restringido a su propio expediente

---

## precondiciones

- el participante debe existir dentro de un proyecto
- debe haberse creado el registro base del expediente al alta del participante

---

## flujo principal

1. El usuario abre el detalle de un participante.
2. El sistema carga su expediente completo.
3. El expediente muestra un resumen general de estado.
4. El usuario puede navegar entre secciones: datos, documentos, contrato, observaciones, pagos e historial.
5. El usuario utiliza el expediente como punto central de operación y consulta.

---

## flujos alternos y excepciones

### expediente incompleto

1. El usuario abre el expediente.
2. El sistema detecta pendientes documentales o contractuales.
3. El encabezado del expediente lo indica de forma visible.

### expediente listo para la siguiente etapa

1. El usuario abre el expediente.
2. El sistema detecta que ya se cumplieron todos los requisitos de una etapa.
3. Muestra el expediente como listo para contrato, pago o cierre según corresponda.

---

## reglas de negocio asociadas

- todo participante debe tener un expediente dentro del proyecto
- el expediente no es un archivo único, sino una agregación estructurada de información y evidencias
- el expediente debe reflejar el estado real más reciente del participante
- los cambios en documentos, contratos o pagos deben reflejarse automáticamente en el expediente

---

## cómo se mide este requisito en el sistema

El requisito se considera cumplido si:

- el usuario puede conocer la situación completa de un participante desde una sola pantalla de expediente
- el expediente muestra información actualizada sin depender de revisar correos o carpetas externas manualmente
- el expediente permite identificar el siguiente paso operativo sin ambigüedad

---

## cómo se traduce en el sistema

### en el frontend

- existe una vista de expediente con resumen superior y secciones internas
- existe una línea de tiempo o historial visible de eventos
- existen indicadores visuales de completitud, observación y etapa actual

### en el backend

- existe un servicio que consolida información de varias entidades para armar el expediente
- el endpoint del expediente devuelve una vista agregada, no solo una tabla aislada

### en la base de datos

- el expediente se apoya en entidades ya existentes: participante, documentos, contrato, observaciones, pagos y auditoría
- puede existir una tabla `expedientes` como entidad lógica o una vista consolidada según el diseño final

---

## cómo se evidencia que se cumple

- la secretaría puede abrir una sola pantalla y conocer qué le falta al participante, qué pasó con su contrato y si ya puede pasar a pago
- el participante puede consultar su propio expediente con acceso limitado
- el expediente cambia automáticamente cuando cambia el estado de cualquiera de sus componentes

---

## cómo se representa en diseño

### en casos de uso

Debe existir el caso de uso "consultar expediente digital".

### en la interfaz

El expediente debe representarse como una vista central de detalle con secciones internas.

### en la arquitectura

Debe existir un servicio agregador de expediente que concentre datos de varios módulos o submódulos.
