# RF_10 — Carga, asociación y consulta de documentos del expediente

## por qué existe este requisito en este sistema

El sistema no solo necesita saber que un documento es requerido. Necesita poder relacionar el archivo real con el expediente correcto. Hoy los archivos viven en OneDrive, se suben, se mueven, se comentan y en ocasiones se pierden en carpetas mal organizadas. Esa es una de las causas principales del desorden operativo.

Por eso, además del control de requisitos, el sistema debe gestionar la referencia del documento real: qué archivo es, a qué participante pertenece, a qué tipo de documento responde, cuál es su estado y cuál fue su última versión válida.

---

## descripción del requisito

El sistema debe permitir cargar, asociar, consultar y versionar documentos vinculados al expediente de un participante o al proyecto.

El sistema debe permitir:

- cargar un documento nuevo al expediente
- asociarlo a un tipo de documento requerido o libre
- consultar el archivo desde el expediente
- reemplazar o subir una nueva versión cuando exista corrección
- marcar el documento como recibido, en revisión, observado, aprobado o rechazado
- identificar quién subió el documento y cuándo

---

## actores involucrados

- secretaría
- participante, cuando sube o consulta documentos permitidos
- revisor interno

---

## precondiciones

- debe existir un expediente o proyecto al cual vincular el documento
- debe existir permiso para cargar o consultar archivos
- el repositorio documental institucional debe estar disponible

---

## flujo principal

1. El usuario abre el expediente.
2. Selecciona el requisito documental correspondiente o la opción de adjuntar archivo.
3. Elige el archivo a cargar.
4. El sistema valida formato, tamaño y asociación.
5. El archivo se guarda en el repositorio institucional y se registra la referencia en el sistema.
6. El documento aparece inmediatamente en el expediente con su estado inicial.

---

## flujos alternos y excepciones

### nueva versión del documento

1. Un documento fue observado.
2. El usuario sube una nueva versión.
3. El sistema conserva la referencia histórica y marca la nueva como versión actual.

### documento no asociado a requisito existente

1. El usuario carga un archivo informativo o complementario.
2. El sistema permite registrarlo como documento adicional con categoría libre.

### formato inválido

1. El usuario intenta subir un archivo no permitido.
2. El sistema rechaza la carga e informa el motivo.

---

## reglas de negocio asociadas

- todo documento debe quedar asociado a un expediente, proyecto o elemento operativo válido
- un documento observado puede requerir nueva versión sin perder trazabilidad
- el documento vigente debe distinguirse de sus versiones históricas
- la carga debe respetar formatos y políticas definidas por la operación

---

## cómo se mide este requisito en el sistema

El requisito se considera cumplido si:

- los documentos pueden abrirse desde el expediente correcto
- las nuevas versiones no destruyen el historial anterior
- el estado documental se actualiza al subir, revisar o aprobar archivos
- el usuario puede identificar fácilmente qué archivo es el vigente

---

## cómo se traduce en el sistema

### en el frontend

- existe componente de carga de archivos dentro del expediente
- existe lista de documentos con estado, fecha y versión
- existe acceso para abrir el archivo o copiar su referencia

### en el backend

- existe integración con SharePoint o OneDrive vía Graph API
- el backend registra metadatos del archivo y su relación con el expediente
- existe lógica de versionado y estados documentales

### en la base de datos

- existe una tabla `documentos`
- existe referencia al identificador del archivo en SharePoint
- existen campos de versión, estado y vínculo con requisito documental

---

## cómo se evidencia que se cumple

- un documento cargado aparece en el expediente sin intervención manual adicional
- el archivo correcto puede abrirse desde el sistema
- al subir una corrección, la versión vigente cambia y la anterior permanece en historial
- el checklist documental reacciona a la carga o aprobación del archivo

---

## cómo se representa en diseño

### en casos de uso

Debe existir el caso de uso "cargar documento" y "consultar documento asociado".

### en el modelo de datos

La entidad `Documento` debe relacionarse con `Expediente`, `TipoDocumento` y `Versión`.

### en la arquitectura

Debe representarse la integración entre el sistema y el almacenamiento documental institucional.
