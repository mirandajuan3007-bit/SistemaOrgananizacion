# RF_07 — Registro de observaciones, correcciones y versiones documentales

## por qué existe este requisito en este sistema

El cliente mencionó de forma explícita que hoy las observaciones al contrato o a los documentos se hacen sobre archivos Word, con comentarios, intercambios por correo y múltiples idas y vueltas que terminan siendo tediosas y difíciles de controlar. Ese dolor no es secundario: es uno de los núcleos del problema actual.

Por eso, el sistema necesita tratar la observación como una entidad operativa formal y no como un comentario informal perdido en un archivo o en un correo.

---

## descripción del requisito

El sistema debe permitir registrar observaciones asociadas a contratos o documentos, controlar su atención y vincular versiones sucesivas del material corregido.

El sistema debe permitir:

- registrar una observación con descripción clara
- asociarla a un contrato o documento específico
- clasificarla por tipo o severidad si se requiere
- marcarla como pendiente, atendida, validada o descartada
- relacionar la observación con la nueva versión del documento o contrato corregido
- mantener historial de observaciones y respuestas

---

## actores involucrados

- secretaría
- responsable revisor
- participante, cuando deba atender observaciones indirectamente

---

## precondiciones

- debe existir un contrato o documento activo sobre el cual operar
- el usuario debe contar con permiso para registrar o resolver observaciones

---

## flujo principal

1. El usuario revisa un contrato o documento.
2. Detecta un problema o ajuste requerido.
3. Registra una observación en el sistema.
4. El sistema vincula esa observación al elemento revisado.
5. El responsable atiende la observación y sube o registra la corrección.
6. El revisor valida la corrección.
7. La observación cambia a estado resuelta o permanece abierta si aún no cumple.

---

## flujos alternos y excepciones

### múltiples observaciones simultáneas

1. Un mismo contrato recibe varias observaciones.
2. El sistema las muestra de manera separada y con estado individual.
3. El contrato solo puede considerarse libre de observaciones cuando todas las críticas estén resueltas según regla definida.

### nueva observación sobre una versión corregida

1. Se revisa una nueva versión.
2. Aún persisten problemas o aparecen nuevos.
3. El sistema permite registrar nuevas observaciones sin borrar las anteriores.

### observación descartada

1. Se determina que una observación no aplica.
2. Se marca como descartada con justificación.

---

## reglas de negocio asociadas

- toda observación debe estar asociada a un elemento concreto
- una observación no puede cerrarse sin una acción de resolución o justificación
- las versiones nuevas no deben sobrescribir el contexto de las versiones previas sin conservar referencia histórica
- el contrato o documento puede depender del estado de sus observaciones para avanzar en el flujo

---

## cómo se mide este requisito en el sistema

El requisito se considera cumplido si:

- cada observación puede consultarse con su estado e historial
- el usuario puede distinguir qué versión respondió a qué observación
- las observaciones abiertas impactan el estado general del expediente o contrato cuando corresponde
- el sistema evita perder observaciones en canales externos no controlados

---

## cómo se traduce en el sistema

### en el frontend

- existe una sección de observaciones por contrato o documento
- las observaciones se listan con estado, fecha, responsable y detalle
- existe acceso a versiones o archivos relacionados a la corrección

### en el backend

- existe un servicio de observaciones y resolución
- el sistema puede relacionar observaciones con archivos o eventos posteriores
- existen reglas para recalcular estados del contrato cuando hay observaciones abiertas

### en la base de datos

- existe una tabla `observaciones`
- existe relación con `contratos`, `documentos` y opcionalmente `versiones_documentales`

---

## cómo se evidencia que se cumple

- al observar un contrato, el sistema obliga a registrar la observación formalmente
- puede verse la lista completa de observaciones abiertas y resueltas
- las correcciones no borran la historia previa
- el expediente refleja si aún existen observaciones pendientes

---

## cómo se representa en diseño

### en casos de uso

Debe existir el caso de uso "registrar observación" y "atender observación".

### en el modelo de datos

La entidad `Observación` debe relacionarse con la entidad revisada y con su estado de resolución.

### en diagramas de flujo

Debe existir el ciclo revisar → observar → corregir → validar.
