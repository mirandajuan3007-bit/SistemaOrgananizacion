# RF_01 — Autenticación institucional y acceso al sistema

## por qué existe este requisito en este sistema

El sistema no es una herramienta pública ni abierta. Opera sobre información institucional, expedientes de personas, contratos, pagos y documentación sensible. Además, los usuarios reales del sistema ya existen dentro de la universidad: secretaría, dirección, investigadores, y eventualmente administradores centrales.

Por esa razón, el acceso no debe resolverse con usuarios y contraseñas locales creadas manualmente dentro de la aplicación. Eso duplicaría la gestión de identidades de la universidad, aumentaría el riesgo de cuentas olvidadas o inseguras, y volvería más costosa la operación diaria.

Lo correcto para este sistema es que cada usuario entre con su cuenta institucional de Microsoft, porque esa cuenta ya forma parte de la infraestructura real en la que vive la operación: Outlook, OneDrive, SharePoint y Microsoft 365. Así, el sistema queda alineado con el ecosistema que ya usa la UADY.

Este requisito es funcional porque el usuario sí realiza una acción concreta: entrar al sistema, identificarse y obtener acceso a las funciones que le corresponden.

---

## descripción del requisito

El sistema debe permitir que los usuarios ingresen mediante su cuenta institucional de Microsoft y, una vez autenticados, deben poder entrar a la unidad académica y a los módulos que les correspondan según su rol.

El sistema debe:

- redirigir al usuario al inicio de sesión institucional cuando intenta acceder sin sesión activa
- validar que la cuenta autenticada pertenece a un usuario autorizado dentro del sistema
- crear el registro local del usuario si es su primer acceso autorizado
- cargar su rol, su unidad académica y los módulos habilitados para esa unidad
- mostrar la pantalla inicial adecuada según su perfil operativo

---

## actores involucrados

- secretaría
- director o responsable académico
- investigador o participante
- administrador técnico del sistema

---

## precondiciones

- el usuario debe tener una cuenta institucional válida en Microsoft Entra ID
- el usuario debe estar dado de alta o autorizado en el sistema
- la unidad académica del usuario debe existir en la configuración interna

---

## flujo principal (esto se hara con un diagrama de flujo conpetentente para marcar los flujos de las funcionalidadesB)

1. El usuario abre la aplicación web.
2. El sistema detecta que no existe una sesión activa
3. El sistema redirige al usuario al proveedor institucional de autenticación de Microsoft.
4. El usuario ingresa con su correo institucional y completa el proceso de autenticación.
5. Microsoft retorna al sistema un token válido de autenticación.
6. El backend valida el token y obtiene el identificador institucional del usuario.
7. El sistema busca al usuario en su base de datos local.
8. Si el usuario existe y está activo, el sistema carga su rol, su unidad y sus permisos.
9. El sistema muestra la pantalla inicial correspondiente a ese usuario.

---

## flujos alternos y excepciones

### usuario autenticado en Microsoft pero no autorizado en el sistema

1. El usuario logra autenticarse con Microsoft.
2. El sistema no encuentra un registro activo para esa cuenta.
3. El sistema muestra un mensaje indicando que la cuenta no tiene acceso habilitado al sistema.
4. El acceso queda bloqueado hasta que un administrador o responsable lo autorice.

### usuario inactivo o suspendido

1. El usuario se autentica correctamente.
2. El sistema encuentra el registro, pero con estado inactivo.
3. El sistema impide el ingreso y muestra un mensaje de acceso suspendido.

### fallo temporal del proveedor de autenticación

1. El usuario intenta iniciar sesión.
2. Microsoft no responde o retorna error.
3. El sistema informa que no fue posible autenticar en ese momento y sugiere reintentar.

---

## reglas de negocio asociadas

- el sistema no debe almacenar contraseñas locales de usuarios, este es importante, para que no haya fugas de informacion
- un usuario solo puede acceder a las unidades y módulos para los que está autorizado
- si un usuario pertenece a más de una unidad académica, debe poder seleccionar su contexto de trabajo al entrar o cambiarlo dentro de la aplicación
- si la cuenta institucional deja de ser válida, el sistema no debe permitir el acceso aunque el usuario exista localmente
- el acceso inicial solo puede darse a usuarios previamente autorizados por la operación del sistema

---

## cómo se mide este requisito en el sistema

El requisito se considera cumplido si:

- el 100% de los accesos al sistema pasa por autenticación institucional
- ningún usuario puede entrar con credenciales locales propias de la aplicación
- un usuario autorizado puede iniciar sesión y llegar a su pantalla inicial en un solo flujo continuo
- un usuario no autorizado es bloqueado sin ver información interna del sistema
- el tiempo entre autenticación exitosa y carga de la pantalla inicial se mantiene en un rango operativo aceptable

---

## cómo se traduce en el sistema

### en el frontend

- existe una pantalla pública mínima de acceso
- si no hay sesión, la aplicación redirige automáticamente al login institucional
- al volver autenticado, el frontend carga el perfil del usuario y el contexto de unidad
- la barra de navegación y los módulos visibles cambian según el rol y la unidad

### en el backend

- existe un módulo de autenticación que valida tokens emitidos por Microsoft Entra ID
- el backend asocia la identidad institucional con un usuario local del sistema
- al iniciar sesión, el backend retorna un perfil operativo con rol, unidad activa y módulos disponibles

### en la base de datos

- existe una tabla de usuarios con el identificador institucional, correo, nombre, rol y estado
- existe relación entre usuario y unidad académica
- existe un campo de activación para bloquear acceso sin borrar el historial del usuario

---

## cómo se evidencia que se cumple

- al abrir la aplicación sin sesión, el sistema redirige al login institucional
- al autenticarse un usuario autorizado, la aplicación carga correctamente su inicio
- al autenticarse un usuario no autorizado, el sistema bloquea el acceso con un mensaje claro
- al desactivar un usuario desde administración, ese usuario ya no puede ingresar
- el historial de acceso registra que el ingreso se realizó con identidad institucional válida

---

## cómo se representa en diseño

### en casos de uso

Debe existir el caso de uso "iniciar sesión con cuenta institucional" con los actores secretaría, director, investigador y administrador.

### en el diagrama de arquitectura

Debe representarse la relación entre frontend, backend y Microsoft Entra ID, indicando que la autenticación se delega al proveedor institucional.

### en el modelo de datos

Debe existir la entidad `Usuario` relacionada con `Unidad`, `Rol` y estado de activación.

### en la navegación

Debe existir una pantalla de acceso y una decisión de enrutamiento inicial según el perfil operativo del usuario.
